# AI 产品 PRD 写作框架（项目宪法）

> 本文件是 InterviewCopilot 项目的 PRD 写作规范。任何 PRD 草稿都必须按此五步法结构化。
> 在 Claude Code 中运行 `/grill-me` 或 `/grill-with-docs` 时，请引用本文件作为追问框架。

---

## 核心理念

### 传统产品 PRD vs AI 产品 PRD（三大差异）

| 维度 | 传统 PRD | AI 产品 PRD |
|---|---|---|
| 核心目标 | 定义**确定性**的流程与交互。输入输出可预测。 | 定义**概率性**的智能体行为。追求"最优解"而非"唯一解"。 |
| 工作流设计 | 用户操作流程图（用户-系统二元交互） | 智能体协作工作流。明确区分"人决策"和"Agent 执行"环节。 |
| 验收标准 | 二元通过性测试（通过/不通过） | 概率性指标评估（如专业度评分 > 4/5，NDCG@10 > 0.85） |

---

## 五步法

```
第一步：用户旅程 → Agent 工作流
   ↓
第二步：功能清单 → 功能点 + 实现要点 + 关键技术
   ↓
第三步：功能详述 → 角色定义 + 任务描述 + 思维链（COT）+ 示例 + 异常处理
   ↓
第四步：数据驱动 → 数据类型 + 规模要求 + 标注标准
   ↓
第五步：测试验收 → 概率性指标 + 场景化测试
```

---

## 第一步：用户旅程 → Agent 工作流

### 核心公式

```
工作流 = 将军（人）+ 士兵（Agent）
```

### 必须回答的问题

1. **哪些环节由"将军（人）"决策？**
   - 例如：用户选择回答风格、用户审核 AI 回答
2. **哪些环节由"士兵（Agent）"执行？**
   - 例如：语音转录、问题分类、回答生成、简历解析
3. **Agent 之间如何传递信息？**
   - 例如：转录 Agent → 回答生成 Agent，传递的是结构化的问题文本

### 输出物

一张协作地图（流程图），清晰标注每个节点是"人决策"还是"Agent 执行"，以及它们之间的信息流向。

---

## 第二步：功能清单 → 功能点 + 实现要点 + 关键技术

### 核心公式

```
AI 功能点 = Agent 任务 = 实现要点 + 关键技术
```

### 表格结构（必填）

| 一级功能 | 二级功能 | 实现要点 | 关键技术/参数 |
|---|---|---|---|
| （Agent 能力模块） | （具体任务） | （怎么做） | （用什么、参数多少） |

### 关键技术参数必须明确

- 支持的文件类型、大小限制？
- 使用哪种模型？（如 Claude Sonnet 4.6、Whisper-1）
- 关键参数？（如 chunk size、top_k、temperature、max_tokens）
- API 调用频率与成本预估？

---

## 第三步：功能详述 → 角色定义 + 任务描述 + 思维链 + 示例 + 异常处理

### 核心公式

```
高质量提示词 = 角色定义 + 任务描述 + 思维链（COT）+ 示例（Few-Shot）+ 异常处理
```

### 每个 Agent 必须给出完整 prompt 模板

例如「STAR 回答风格 Agent」的 prompt：

```
角色定义：你是一位资深面试教练，擅长用 STAR 结构帮助求职者
         结构化表达自己的经历。

任务描述与 COT：
   1. 首先识别面试官问题的类型（行为/技术/情境）
   2. 从用户简历中检索最匹配的 1-2 段真实经历
   3. 按 Situation → Task → Action → Result 套入
   4. 确保 Result 部分有可量化的数字

Few-Shot 示例：
   示例 1：[问题] → [STAR 回答]
   示例 2：[问题] → [STAR 回答]

异常处理：
   - 当简历中找不到相关经历时：用"假设性回答 + 学习意愿"格式
   - 当问题过于宽泛时：礼貌追问以缩小范围
```

### 语言精准

- 不啰嗦，不惜字如金
- 背景信息完整
- 指令清晰无歧义

---

## 第四步：数据驱动 → 数据类型 + 规模要求 + 标注标准

### 核心公式

```
高质量输出 = 高质量数据 × 优秀算法
```

### 必须问 What 和 Why

- **What（需要什么数据）**：为了训练或评估模型，需要哪些数据集？
- **Why（为什么需要）**：用于模型微调？作为测试黄金标准？还是 RAG 上下文？

### 表格结构

| 数据类型 | 描述与用途 | 规模与要求 | 注意事项 |
|---|---|---|---|
| 测试问题集 | 用于评估 AI 回答质量 | 30 组高频面试题 | 涵盖行为/技术/情境/HR 四类 |
| 简历样例 | 用于测试简历解析准确率 | 10 份不同行业 PDF | 覆盖 1 页/2 页/扫描件 |
| 黄金答案 | 由 PM 标注的"理想 STAR 回答" | 30 组 | 用于盲评对照 |

---

## 第五步：测试验收 → 概率性指标 + 场景化测试

### 核心转变

```
确定性规则 → 概率性指标
```

### 必须回答 How

- 如何设计测试用例？
- 用什么可量化的指标来衡量效果好坏？

### 表格结构

| 功能点 | 测试方法 | 概率性指标（How to Measure） |
|---|---|---|
| STAR 回答生成 | 专家盲评 | 专业度、结构完整性 > 4 分（5 分制） |
| 简历解析准确率 | 与人工标注对照 | 关键字段提取准确率 > 90% |
| 端到端响应延迟 | 自动化压测 | 从问题转录完到首字输出 < 2 秒，P95 < 3 秒 |
| 触发时机准确度 | 人工评测 | 误触发率 < 10% |

## 三大核心心法

### 1. 结构化思维（SOP）
将 PRD 写作流程化，将功能清单分层级（一级/二级功能）。

### 2. 批判性思维（辩证看待）
不盲目照搬竞品或技术方案，始终思考：
- "**是否合理？**"
- "**如何本土化适配？**"

### 3. 闭环思维（What-Why-How）
在 PRD 的每个环节，下意识地追问：
- 我们需要什么？（What）
- 为什么需要它？（Why）
- 最终如何实现和验证它？（How）

## 在 Claude Code 中如何使用本框架

### 推荐流程

```
1. /grill-with-docs
   → 引用本文件，让 agent 按五步法逐步追问

2. 完成 5 轮追问后，agent 会有完整的草稿信息

3. /to-prd
   → 把对话内容沉淀为正式 PRD（保存到 docs/PRD.md）

4. /to-issues
   → 把 PRD 拆成 GitHub Issues，每个 issue 都是一个可独立完成的任务
```

### 关键指令模板

每次启动 grill 时，提示 agent：

> "请按 PRD-FRAMEWORK.md 中的五步法追问我。
> 不要跳到下一步，除非当前步骤的所有问题都已回答清楚。
> 你是士兵，我是将军，但当我的指令模糊时，你必须追问让我明确。"
