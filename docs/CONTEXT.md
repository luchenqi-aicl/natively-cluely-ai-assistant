## 项目定位

- **底座**：基于 Natively（Electron + React + Rust，v2.6.0）改造

## 第一步结论：用户旅程 → Agent 工作流

### 核心公式

```
工作流 = 将军（人决策）+ 士兵（Agent 执行）
```

### 三个阶段

#### 阶段一：面试前准备

| 节点 | 类型 | 说明 |
|---|---|---|
| 上传简历（PDF/DOCX） | 将军决策 | 用户主动上传 |
| 粘贴目标职位 JD | 将军决策 | 用户主动粘贴 |
| 选择面试类型 | 将军决策 | 技术面 / HR 面 / 行为面 / 综合 |
| 设置输入语言 + 输出语言 | 将军决策 | 输入语言 = 面试官说话语言；输出语言 = 提词显示语言；默认中文 |
| 简历解析器 | Agent 执行 | 提取结构化经历、技能、项目 |
| JD 解析器 | Agent 执行 | 提取岗位要求、关键词 |

解析结果存入「会话上下文」，供 Answer Generator 全程调用。

#### 阶段二：面试中核心循环

| 节点 | 类型 | 说明 |
|---|---|---|
| 系统音频捕获 → STT 实时转录 | Agent 执行 | 仅捕获面试官声道（系统音频） |
| 静音超过 1.5 秒 → 自动触发 | 自动触发 | 面试官停止说话后自动生成提词 |
| 用户按「c」键 → 手动触发 | 将军触发 | 用户主动控制，兜底机制 |
| 问题分类器 | Agent 执行 | 判断问题类型：行为类 / 非行为类 |
| Answer Generator（行为类） | Agent 执行 | 输出 STAR 框架 + 部分预填 + 关键词加粗 |
| Answer Generator（非行为类） | Agent 执行 | 输出完整答案草稿 + 关键词加粗 |
| 悬浮窗展示提词 | Agent 执行 | 用目标输出语言显示 |
| 用户读取提词、组织语言、作答 | 将军决策 | 人始终掌控最终表达 |

#### 阶段三：面试后复盘

| 节点 | 类型 | 说明 |
|---|---|---|
| 复盘总结器 | Agent 执行 | 自动生成：本场问题清单 + 每题 AI 提词要点摘要 |
| 用户查阅复盘报告 | 将军决策 | — |

### Agent 信息流

```
简历/JD ──────────────→ Parser ──→ 结构化上下文 ──────────────────┐
                                                                  ↓
面试官语音 → STT → 问题文本 → 分类器 → 类型标签 → Answer Generator
                                                    ↑             ↓
                                               语言设置      格式化提词
                                                         （STAR 或全文草稿，关键词加粗）
全程转录 + 提词记录 ──────────────────────────→ 复盘总结器 → 复盘报告
```

### 术语定义

| 术语 | 定义 |
|---|---|
| 提词 | AI 生成的实时面试辅助文本，在悬浮窗中展示 |
| 会话上下文 | 本场面试开始前加载的简历结构化数据 + JD 关键词，供 Agent 全程调用 |
| 行为类问题 | 以「请描述一次…」「举个例子…」「Tell me about a time…」等句式开头的问题，触发 STAR 框架输出 |
| STAR 框架 | Situation / Task / Action / Result 四段式回答结构 |
| 双触发机制 | 静音 1.5 秒自动触发 ＋ 用户按「c」键手动触发，二者均可启动 Answer Generator |
| 输入语言 | 面试官说话的语言，用于 STT 识别配置 |
| 输出语言 | 提词显示的语言，默认中文，可在设置中切换 |
| 复盘报告 | 面试结束后自动生成的总结文档，包含问题清单、AI 提词要点、质量评分和改进建议 |
| 转行故事 | 简历中识别出的跨行业叙事（如「卫生政策 → AI PM」），单独保留为上下文字段，供 Answer Generator 主动引用 |

## 第二步结论：功能清单 → 功能点 + 实现要点 + 关键技术

### 优先级分层

| 优先级 | 模块 | 理由 |
|---|---|---|
| P0 | Answer Generator 提示词工程 | 直接决定提词是否个性化，用户最直接感知的质量 |
| P0 | 简历/JD 解析 → 会话上下文构建 | Answer Generator 的燃料，解析质量决定提词质量 |
| P1 | 复盘总结器 | 真正差异化功能，不影响核心面试流程 |
| 复用 | STT 转录 + 问题分类 + 双触发机制 | Natively 已实现或由模型能力直接决定，改动最小 |

### 功能表

| 一级功能 | 二级功能 | 实现要点 | 关键技术 / 参数 |
|---|---|---|---|
| **会话上下文构建** | 简历解析 | 提取：工作/实习经历（公司/职位/时间/成就）、项目经历（角色/技术/成果）、学历背景、核心技能；识别转行故事单独保留 | `claude-opus-4-7`（BYOK，用户可换）；`pdf-parse` + `mammoth`（已在 Natively）；输出结构化 JSON |
| **会话上下文构建** | JD 解析 | 提取：岗位核心要求、高频关键词、必备技能；与简历字段做初步对齐标注 | 同上；prompt 要求输出 keyword list + requirement list |
| **会话上下文构建** | 转行叙事识别 | 检测简历中跨领域跳跃（如「医疗/公共卫生 → 科技/PM」），生成 1-2 句「转行故事」字段 | 同上；作为独立 JSON 字段存入会话上下文 |
| **实时提词生成** | 双触发机制 | 静音 ≥ 1.5s 自动触发 OR 用户按「c」键手动触发 | Natively 现有静音检测逻辑复用；键盘监听已有 |
| **实时提词生成** | 问题分类 | 判断行为类（STAR）/ 非行为类（全文草稿） | prompt 内嵌分类指令，随 Answer Generator 一次调用完成，不单独请求 |
| **实时提词生成** | Answer Generator | 注入会话上下文（简历摘要 + JD 关键词）+ 问题文本 + 类型标签；行为类输出 STAR 框架（S/T/A/R 四格 + 预填）；非行为类输出完整草稿；关键词加粗（markdown `**`）；输出语言由用户设置决定 | `claude-haiku-4-5`（默认，已集成）；Groq 作备选；目标首字 < 1.5s；`max_tokens` ≈ 300；`temperature` 0.4 |
| **复盘总结器** | 问题清单整理 | 汇总本场所有转录问题，去重、排序 | `claude-opus-4-7`（BYOK）；批处理，面试结束后一次性调用 |
| **复盘总结器** | 提词要点归纳 | 每题对应的 AI 提词要点压缩为 2-3 句摘要 | 同上 |
| **复盘总结器** | 质量评分 | 对每题评分，维度：① STAR 结构完整度 ② JD 关键词对齐度 ③ 具体性（有无数字/案例）④ 转行叙事连贯性（转行用户专项） | 评分 1-5 分制；prompt 包含评分标准示例（Few-Shot） |
| **复盘总结器** | 改进建议 | 针对低分项给出具体改进方向，不超过 2 条/题 | 同上；引用面试评估领域 prompt 框架（待调研） |
| **语言设置** | 输入/输出语言配置 | 输入语言（STT 识别语言）和输出语言（提词显示语言）分开设置；默认均为中文 | Natively 现有 STT 语言参数复用；Answer Generator prompt 末尾注入语言指令 |

## 第三步结论：功能详述 → Answer Generator Prompt 模板

### 角色定义

```
你是一位精通 AI 产品和求职面试的实战教练，专门为跨行业转型、非英语母语的求职者提供即时面试提词。
你熟悉 STAR、三层产品洞察、Skills Bridge 等框架，能在 10 秒内给出结构化、个性化的回答骨架。
```

### 上下文注入（每次调用必须注入）

```
应聘者工作/项目经历：{resume_context}
转行故事：{career_pivot_story}（如有）
JD 核心关键词：{jd_keywords}
面试类型：{interview_type}
输出语言：{output_language}（默认：中文）
输出模式：{output_mode}（auto / skeleton / full）
```

### 任务描述 + 思维链（COT）

```
收到面试官问题后，按以下步骤处理：

① 识别问题类型：
   - 行为类（含「举个例子/描述一次/Tell me about a time」）
     → 用 STAR；若问题含「困难/失败/挑战」→ 改用 SOAR
   - 产品洞察类（含「喜欢什么产品/怎么看这个赛道/评价某产品」）
     → 用三层结构
   - 动机类（含「为什么选/为什么转/自我介绍」）
     → 用过去→现在→未来
   - 其他 → 直接全文草稿

② 从 resume_context 中检索最相关的 1-2 段经历或项目

③ 判断是否激活转行故事：
   若问题涉及「跨背景/动机/独特性」且 career_pivot_story 非空
   → 主动引用转行故事字段

④ 按框架生成提词，并对齐 ≥1 个 jd_keywords，对齐词加粗

⑤ Result/成果部分：必须引用 resume_context 中的真实数字。
   若简历中无对应数字，留占位符 [具体数字] 提示用户补充。
   绝对禁止捏造数据。

⑥ 判断输出长度：
   - output_mode = auto：字数 ≤ 400 → 全文；> 400 → 压缩为骨架
   - output_mode = skeleton：始终输出 150-250 字骨架
   - output_mode = full：始终输出完整草稿

⑦ 用 output_language 输出，关键词/核心论点加粗（markdown **）

风格要求：口语化，说人话。
禁止出现：「首先/其次/最后」「综上所述」「值得一提的是」「不得不说」
          「总而言之」「毋庸置疑」等 AI 套话。
目标：面试官听到的是真实的人在说话，不是在背稿。
```

### Few-Shot 示例

**示例 1｜产品洞察类 → 三层结构**

问题：「你最近最喜欢哪个 AI 产品？」

```
【赛道】持续关注 **AI 社交 / 数字分身** 赛道，以 SecondMe 为例
——比纯工具类更能体现 AI 如何改变人的表达与关系

【亮点 + 问题】
· 打动我的点：**从情绪价值走向信息价值**
  经验可规模化分发（结合求职辅导经历展开）
· 赛道结构性问题：**人格一致性、长期留存**是整个赛道未解难题
· 执行层面问题：场景体验闭环不完整，创意 > 完成度

【趋势钩子】AI 社交正从情绪价值迁移到 **人格资产**，方向尚早但值得关注
```

**示例 2｜行为类 → STAR**

问题：「描述一次你用数据推动决策的经历」

```
**S**：PKU 卫生政策项目，数据结果与团队预期出入较大
**T**：需说服团队采纳数据驱动建议，而非沿用经验判断
**A**：用 **R/Python 重新建模**，可视化呈现差异；主动约谈核心决策方说明方法论
**R**：建议被采纳，报告中 **3 项政策建议**被写入最终方案，
     数据分析方法被团队后续 **2 个项目**复用
     （Result 数字必须来自简历真实内容，无数据时留 [具体数字]）
```

### 异常处理

| 场景 | 处理方式 |
|---|---|
| 问题过于宽泛（如「说说你自己」） | 默认触发过去→现在→未来框架 + 激活转行故事 |
| 简历中无相关经历 | 格式：「我在 [相近经历] 中接触过类似挑战，虽无直接 XX 经验，我的判断是……」|
| 面试官说英文但 output_language = 中文 | 仍用中文输出提词（用户看中文提词、说英文）|
| 问题类型无法识别 | 默认全文草稿模式，不强行套框架 |

### 提词输出模式（用户可在设置中选择）

| 模式 | 行为 |
|---|---|
| **自动模式**（默认） | ≤ 400 字输出完整草稿；> 400 字自动压缩为骨架 |
| **骨架模式** | 始终输出 150-250 字骨架 + 关键论点 |
| **全文模式** | 始终输出完整草稿 |

### 问题类型识别框架（Answer Generator 动态选择）

| 问题类型 | 识别信号 | 输出框架 |
|---|---|---|
| 行为类 | 举个例子 / 描述一次 / Tell me about a time | STAR（含困难/失败 → SOAR）|
| 产品洞察类 | 喜欢什么产品 / 怎么看这个赛道 / 评价某产品 | 三层结构（赛道选择→亮点+两类缺陷→趋势钩子）|
| 动机类 | 为什么选 / 为什么转 / 自我介绍 | 过去→现在→未来 + 转行故事 |
| 其他 | — | 直接全文草稿 |

## 第四步结论：数据驱动 → 数据类型 + 规模要求 + 标注标准

### 数据集计划

| 数据类型 | 描述与用途 | 规模要求 | 注意事项 |
|---|---|---|---|
| 面试题目集 | 测试 Answer Generator 输出质量 | 30 题（行为/产品洞察/动机/技术各 7-8 题），以 AI PM 实习为目标岗位 | 覆盖四类问题类型 |
| 黄金标准答案 | 评分基线，验证复盘评分模块 | 与 30 题一一对应 | AI 生成后**必须人工过滤**：标记「会真正说的」vs「太套路」，只有人工标记「会说」的才算黄金标准 |
| 模拟简历 | 测试简历解析模块的鲁棒性 | ≥ 3 份（转行者 / 留学生 / 应届生） | 至少包含 1 份跨行业背景（如卫生政策 → AI PM）|

### 验证方法

- **MVP 阶段**：人工评分 vs AI 评分对照，验证评分逻辑是否一致
- **上线后**：收集真实用户使用数据作为迭代依据

### 标注标准（1-5 分制）

**① STAR 结构完整度**

| 分 | 标准 |
|---|---|
| 5 | S/T/A/R 四格齐全，Result 有具体数字 |
| 4 | 四格齐全，Result 无数字但有明确结果 |
| 3 | 缺一格，或 Result 只有「被采纳」等笼统表述 |
| 2 | 只有 S+A，缺 T 和 R |
| 1 | 基本没有结构，流水账 |

**② JD 关键词对齐度**

| 分 | 标准 |
|---|---|
| 5 | 主动命中 ≥ 3 个 JD 核心关键词，且自然融入回答 |
| 4 | 命中 2 个关键词 |
| 3 | 命中 1 个，或提到了但很生硬 |
| 2 | 方向沾边但无关键词 |
| 1 | 与 JD 完全无关 |

**③ 具体性（有无真实数字/案例）**

| 分 | 标准 |
|---|---|
| 5 | 有 ≥ 2 个可量化数字或命名具体项目 |
| 4 | 有 1 个数字或具体案例 |
| 3 | 有细节但无数字（「做了很多分析」） |
| 2 | 完全抽象（「我有很强的数据能力」） |
| 1 | 空话套话 |

**④ 转行叙事连贯性**（仅适用于跨行业背景用户）

| 分 | 标准 |
|---|---|
| 5 | 主动将旧领域经历转化为新岗位的直接优势，有具体桥接 |
| 4 | 提到了跨行背景，但桥接较弱 |
| 3 | 只提旧经历，未说明与新岗位的关联 |
| 2 | 回避了跨行背景 |
| 1 | 跨行背景成为减分项（没有解释为什么转）|

## 第五步结论：测试验收 → 概率性指标 + 场景化测试

### 验收标准表

| 功能点 | 测试方法 | 指标（通过线） |
|---|---|---|
| **Answer Generator 提词质量** | 30 题黄金答案人工盲评（1-5分） | 四维度均分 ≥ 3.5 / 5（以逻辑链完整为核准） |
| **实时提词延迟** | 秒表计时：触发到悬浮窗首字出现 | 中位数 < 2s，P95 < 3s |
| **问题类型识别准确率** | 30 题人工标注 vs AI 分类对照 | 准确率 ≥ 85% |
| **简历字段提取准确率** | 3 份模拟简历，人工标注 vs AI 提取对照 | P0 字段（工作/项目/JD关键词）准确率 ≥ 90% |
| **复盘评分一致性** | 同一组答案，人工评分 vs AI 评分 | 平均分差 ≤ 0.8 分（5分制） |
| **双触发机制误触发率** | 模拟 20 次对话，记录误触发次数 | 误触发率 ≤ 10% |

### 核心转变

```
确定性规则（能不能跑起来）→ 概率性指标（跑得好不好）
```

### 模型选型决策

| 场景 | 默认模型 | 理由 | 备选 |
|---|---|---|---|
| 非实时（简历解析、复盘总结） | `claude-opus-4-7` | 无延迟压力，用最强模型保证质量；BYOK 由用户承担成本 | 用户可在设置中自由切换 |
| 实时提词生成 | `claude-haiku-4-5` | 首字 ~0.5-1.0s，中文 STAR 结构遵循质量优于 Groq，已集成 | Groq（速度优先时备选） |
